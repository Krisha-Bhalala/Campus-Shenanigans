System Design
Built in layers.
Stays clean.

Five fully separated layers, each with a single responsibility, communicating only through defined interfaces, testable in complete isolation.

Group 6 — Architecture Diagram
Presentation
Activities · Fragments · ViewModels · Presenters
Logic
AuthService · EventService · Validators · SessionManager
Data Layer
Domain Models · Persistence Interfaces · Repositories
Persistence
SQLite · Stubs · Mappers · Schemas
Application
CampusShenanigans · ServiceContainer · PersistenceContainer

Data Flow — 5 Steps

1
User ActionPresentation layer captures input and delegates requests to the logic layer.
2
Business LogicServices apply validation rules, permissions, and business workflows independent of UI or storage.
3
Data AccessLogic interacts with persistence through interfaces, ensuring loose coupling.
4
Persistence ExecutionConcrete implementations (SQLite or stubs) perform CRUD operations and map data to domain models.
5
UI UpdateProcessed results propagate back to the presentation layer, updating the UI state.

Why This Works

The architecture enforces strict separation of concerns. Business logic remains completely independent of UI and storage, while persistence is interchangeable through interfaces. This allowed the team to switch from stub data to a real SQLite database with zero changes to the logic and presentation layers.
